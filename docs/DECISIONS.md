2026-07-17 — Issue #23's `Enrichment Orchestrator` sets `onError: "continueRegularOutput"` and `alwaysOutputData: true` on all three of its `Execute Workflow` calls (Website Scraper, Tech Stack Detector, News RSS) — both found necessary via live testing, not applied speculatively. Why: without `onError`, a called sub-workflow's own genuine failure (e.g. Tech Stack Detector's `config_unreadable`, see #26) propagated up and killed the entire orchestrator chain, so later steps never ran and the failure double-alerted (once from the sub-workflow's own Error Workflow, once from the orchestrator's). Without `alwaysOutputData`, a called sub-workflow completing successfully but with zero output items (Tech Stack Detector's `Config Unreadable?` IF node has no connection wired for its normal/false branch, by design, since standalone nothing needs to run after it) silently truncates the orchestrator mid-chain with no error and no alert — a recurrence of the #21 RUNBOOK "zero output items" gotcha through a new door, caught live during alert-path verification when a poll for completion never resolved because the execution had already finished, incompletely. Both fixes are cheap and defensive; without them, a sub-workflow that behaves perfectly well standalone can silently break composition the moment it's called from a sequential orchestrator.
